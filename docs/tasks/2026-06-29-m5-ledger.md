# Ledger module — file-level conflict tracking for concurrent Crafters

**Created**: 2026-06-29T12:00:00Z
**Status**: in-progress

## Goal

A lightweight conflict-tracking module that records which files are claimed by in-flight Crafters, and provides a pre-dispatch check so the orchestrator (M6) can hold back Crafter spawns whose target files overlap with an already-running Crafter's claims. This makes concurrent Crafter dispatch safe: independent-file steps run in parallel; overlapping-file steps serialize automatically.

## Non-goals

- ❌ Git-level merge conflict resolution — worktrees handle that (M7 integration)
- ❌ Transactional guarantees or distributed locking — single-process, single orchestrator
- ❌ Persistence across session restart — in-memory only for M5; persistence decision remains an open question for later
- ❌ Dispatching agents — the orchestrator (M6) queries the ledger, the ledger doesn't trigger actions

## Approach

### Data model

```ts
interface LedgerEntry {
  agentId: string;       // which Crafter claimed these files
  files: Set<string>;    // absolute file paths this agent expects to touch
  action: string;        // "edit" | "create" | "delete" | "read+write"
  timestamp: number;     // Date.now() when claimed
}
```

The ledger is a class with a `Map<string, LedgerEntry>` keyed by `agentId`. Operations are synchronous (no async I/O needed for in-memory).

### Pre-dispatch check flow

1. Orchestrator computes `unblockedSteps` from the plan file (M2)
2. For each unblocked step, it estimates which files the step will touch (from step description + Plan's guidance)
3. It calls `ledger.getConflictingFiles(candidateFiles)` → gets the subset already claimed
4. If the subset is non-empty → step waits (even though plan-file deps are satisfied)
5. If the subset is empty → step is clear, dispatches Crafter, calls `ledger.claim(agentId, files)`
6. On Crafter completion (`subagents:completed`) → `ledger.release(agentId)` + re-evaluate

### Why "estimated files" is acceptable

The ledger checks files the step is *expected* to touch based on the Plan agent's output, not what the Crafter *actually* touches (which isn't known until it runs). This is conservative — false conflicts hold back a step unnecessarily but safely; missed conflicts still rely on worktree isolation (M7). The Plan agent's system prompt (M1) instructs it to identify key files per step, which feeds this estimation.

  - Returns the union of all in-flight agents' file sets

## Open design decision

| Decision | Current stance | Trade-off |
|---|---|---|
| **Persistence across interruption** | In-memory only for M5 | Simpler. If session restarts mid-pipeline, the ledger resets and the orchestrator re-evaluates from plan file state. Moving to `pi.appendEntry` persistence later adds resume-after-crash but requires serialization of `Set<string>` → `string[]` and replay logic on load. |
